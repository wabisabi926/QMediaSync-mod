# 注释规范

> 职责：定义 QMediaSync 源码、Swagger 注解和验证代码的注释边界。
>
> 权威范围：本文档维护注释的语言、信息密度和安全边界；业务协议和实现细节以对应源码和契约文档为准。
>
> 修改时机：新增或调整跨模块规则、导出 API、历史兼容逻辑或注释约定时必须更新本文档。
>
> 相关代码：`backend/`、`frontend/src/`。

## 基本原则

- 注释默认使用中文；品牌名、协议名和通用技术名词保留官方英文，例如 API、SSE、STRM、Emby、OpenList、GORM。
- 注释优先说明业务原因、安全边界、兼容背景、状态不变量和第三方限制，而不是逐行复述代码行为。
- 修改代码时同步更新或删除过时注释。无法由当前实现支撑的历史说明不得保留。
- 注释不能替代实现、类型、输入校验、权限校验、脱敏或自动化验证。安全风险优先由代码和验证约束，注释只解释原因和边界。

## Go 注释和 Swagger

- 导出的 Go 类型、函数、方法和跨包变量应有中文文档注释，说明其职责或调用边界；简单私有辅助函数不机械添加注释。
- 结构体字段注释说明业务含义、单位、存储语义或兼容限制；重复字段名的注释不增加维护价值。
- Swagger 注解使用英文标签和中文描述，并与控制器实际响应、字段和鉴权规则保持一致。
- 对事务、重试、幂等、并发、迁移和外部 API 兼容代码，注释应说明不可违反的条件和违反后的风险；具体规则同时更新到对应契约文档。

## Vue 和 TypeScript 注释

- Vue 组件和组合式函数优先通过清晰命名、类型和小函数表达意图。只有无法从代码直接推导的响应式边界、异步取消、可调用参数或兼容原因才添加注释。
- 不为模板中的普通绑定、显而易见的 `ref` 赋值、条件分支或简单请求调用添加复述性注释。
- `MaybeRefOrGetter`、`toValue()`、`MaybeRef` 和 `unref()` 的选择涉及可调用对象时，应在公共组合式函数或关键适配点说明避免误执行 callback / client 的原因。

## 验证代码和夹具

- 包内 Go 测试使用 table-driven 场景时，测试名和 case 名应表达业务场景；业务场景优先使用中文，既有英文测试名保持不变。
- 复杂夹具、Mock 或回归测试应在文件头或紧邻夹具处说明它保护的回归面和业务角色，不解释普通断言、循环或变量赋值。
- 临时排查代码不得通过“临时”“待删除”等注释混入长期源码；应放在本地临时位置，或在同一变更中删除。

## 不写的内容

- “创建对象”“调用函数”“设置属性”“处理错误”等复述下一行代码的注释。
- 装饰性分隔线、表情、编号已经失真的步骤说明，以及“修改开始 / 修改结束”“这里很重要”等无信息标记。
- 不能验证、没有来源或与当前行为矛盾的实现猜测。
- 在注释中记录密钥、Token、Cookie、完整请求凭据或其他敏感数据。

## 审查清单

- 注释是否解释了“为什么”或约束，而不是“做了什么”？
- 注释是否与当前代码、测试和权威文档一致？
- 这个问题是否更适合用类型、校验、测试或契约文档解决？
- 修改是否同步更新了对应 Swagger、测试说明或正式文档？

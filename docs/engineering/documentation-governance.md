# 文档治理

> 职责：定义面向仓库维护者和 AI 的正式文档分层、命名和维护规则。
>
> 权威范围：本文件是正式文档位置、职责边界和迁移方式的唯一约定；具体业务事实仍以其所属的权威文档和源码为准。
>
> 修改时机：新增、拆分、移动正式文档，或发现多篇文档重复描述同一事实时，必须同步更新本文件和 [文档索引](../README.md)。
>
> 相关入口：[仓库协作约束](../../AGENTS.md)、[文档索引](../README.md)。

## 目标

文档服务于仓库维护和 AI 辅助代码生成。读者应能在修改前快速确定：

- 哪篇文档定义当前改动涉及的行为或数据契约；
- 该事实是否已有唯一的权威位置；
- 修改完成后必须同步哪些文档；
- 哪些信息应留在代码附近，而不应重复到全局文档。

正式文档记录稳定的边界、协议、数据语义和维护流程，不充当源码导览或提交记录。实现细节以源码为准；只有跨模块、容易被误解或无法从代码直接推导的事实才需要进入正式文档。

## 文档层级

| 位置 | 职责 | 适用内容 |
| --- | --- | --- |
| `AGENTS.md`、`CLAUDE.md` | AI 编码工具的兼容入口 | 相同的硬规则和完整说明入口；不重复项目事实和细节约定 |
| `docs/README.md` | 正式文档唯一索引 | 按职责列出文档及一句话说明 |
| `docs/architecture/` | 跨模块系统契约 | 事件、同步链路、状态流转、组件边界和一致性要求 |
| `docs/engineering/` | 开发和维护约定 | AI 协作、本地开发、前端实现、校验、验证、仓库结构和文档治理 |
| `docs/operations/` | 运行和发布维护 | 部署与持久化、配置、数据库运维、反向代理、发布和恢复流程 |
| `docs/reference/` | 精确查阅资料 | API、枚举、字段语义、配置字段和兼容性表 |
| 代码目录内的 `README.md` | 局部模块说明 | 包职责、公开入口、限制和与相邻模块的关系 |

`docs/examples/` 仅存放可复制的示例文件。

## AI 编码工具入口

`AGENTS.md` 和 `CLAUDE.md` 是等价的兼容入口，文件内容必须完全相同，并且只保留以下内容：

- 必须先阅读 `docs/engineering/ai-assistant.md`；
- 修改前必须通过 `docs/README.md` 定位相关权威文档，同时阅读相关实现和测试；
- 保持改动聚焦、保护用户未提交修改，并按改动范围运行验证；
- 新增、移动或更新正式文档时遵循本文档。

完整、长期维护的 AI 协作规则只维护在 `docs/engineering/ai-assistant.md`。该文档应包含项目快照、后端和前端开发约定、最小验证命令、文档同步映射、局部 README 的使用边界，以及链接到专题契约文档的导航。

不要在两个入口文件中分别维护完整规则，也不要仅因工具名称不同增加不同的业务约束。新增 AI 编码工具兼容入口时，优先让其指向同一份完整说明；只有工具确实不支持相同格式时，才补充最小适配说明。

## 职责边界

每个可独立维护的事实只能有一个权威文档。其他文档只说明自身边界，并使用链接跳转到权威位置。

例如，STRM Webhook 的请求和响应字段由 `reference/strm-webhook.md` 维护；上传与 STRM 后处理的状态流转由 `architecture/upload-and-strm-processing.md` 维护；配置文档只说明如何启用该能力，不复制接口字段表。

下列信息不应写入全局正式文档：

- 可直接从单一函数或类型定义得出的局部实现；
- 仅服务某次改动的设计草案、执行计划和排查过程；
- 已失效的历史实现说明。

它们分别应留在代码注释或 Git 历史中。

## 命名规则

- 正式 Markdown 文件使用小写 `kebab-case` 英文名；目录索引使用 `README.md`。
- 文件名描述稳定的业务概念或文档职责，不使用临时实现、UI 名称或某次改动的动词短语。
- 标题使用中文，首次出现的协议、产品和通用技术名词保留官方英文，例如 SSE、STRM、Emby、OpenList。
- 目录和文件名不使用不必要的缩写；`api`、`sse`、`strm` 等通用缩写可以保留。
- 同一概念在文件名、标题、索引描述和代码注释中使用相同术语。

推荐示例：`realtime-events.md`、`request-validation.md`、`emby-library-sync.md`。

不推荐示例：`realtime-sync-task-refresh.md`（绑定具体刷新结果）、`development.md`（职责过宽）、`new-api.md`（无法说明长期范围）。

## 正式文档模板

除 `docs/README.md` 和代码目录内的简短 `README.md` 外，正式文档开头应包含以下元信息：

```md
> 职责：说明本文档回答的问题。
>
> 权威范围：说明本文档唯一维护的事实，以及不覆盖的边界。
>
> 修改时机：列出哪些代码、配置或流程变化必须同步更新。
>
> 相关代码：`backend/...`、`frontend/...`
```

元信息应简短、可验证，并指向具体目录或模块；不使用“按需更新”“相关代码很多”等模糊描述。

## 契约文档

契约文档用于维护跨模块、跨端或外部接口的高风险不变量。普通概念说明和局部模块 README 不必机械增加契约字段。

QMediaSync 当前应优先建立或强化以下契约：

- API 响应、请求绑定和兼容边界；
- 配置优先级、敏感数据和本机加密密钥；
- 数据库迁移、时间字段和持久化语义；
- 实时事件的快照、回放和断线恢复边界；
- 同步、上传和 STRM 任务的状态流转与幂等边界；
- Emby 刷新与同步的入口、状态字段和触发边界。

除通用元信息外，契约文档还应包含：

```md
## 不变量

- 不得改变的行为、数据语义或安全边界。

## 验证方式

- 对应的测试或最小命令，以及它保护的回归面。
```

不变量必须可以映射到具体代码或可观察行为；验证方式必须说明运行命令或测试位置。不能由现有自动化验证覆盖时，应明确人工检查步骤和剩余风险，而不是只写“注意兼容”。

## 验证说明

`docs/engineering/verification.md` 统一维护验证命令的选择方式，以及“改动类型 → 最小验证”的映射。

- 后端的稳定回归验证优先使用相应包内、可提交的 Go 测试。
- 当现有验证无法覆盖明确的长期回归风险时，优先补充对应包内测试；无法自动覆盖时，在契约文档中写明人工检查步骤和剩余风险。

## 维护流程

1. 修改前，从 `AGENTS.md` 或 `CLAUDE.md` 阅读 `docs/engineering/ai-assistant.md`，再通过 `docs/README.md` 定位与改动职责一致的权威文档、相关代码和测试。
2. 发现一个事实在多处重复时，选择职责最匹配的一处作为权威位置；其余位置改为简短说明和链接。
3. 修改高风险边界时，先确认是否存在对应契约；没有时补充契约文档的不变量和验证方式。
4. 新增文档前，先确认它无法归入现有文档；可以归入时优先扩充现有文档。
5. 新增或移动正式文档时，更新 `docs/README.md`，并确认仓库内旧链接和旧名称已更新。
6. 完成后检查文档元信息、内部链接、术语和文件名；不把本地计划或临时记录加入正式索引。

## 新增或迁移原则

文档层级调整不与业务功能改动混在同一个变更中。新增或迁移时按以下顺序执行：

1. 确认内容职责和唯一权威位置；可以归入已有文档时优先扩充已有文档。
2. 创建或移动文件后，为正式文档补充元信息；高风险跨模块文档补充不变量和验证方式。
3. 更新 `docs/README.md`、AI 入口、代码目录 README 和仓库内 Markdown 链接。
4. 确认旧路径不再被正式文档或源码引用，Markdown 链接目标存在；修改 AI 入口时，确认 `AGENTS.md` 与 `CLAUDE.md` 完全一致。

不保留内容重复的旧路径兼容页，以避免维护者和 AI 误把旧页视为权威来源。需要对外说明路径变化时，在对应发布说明中简短记录。
